# Testing React Applications That Sit Next to Real Devices

Tests are the safety net that let me change a dashboard or operator panel without breaking the controls that matter. The goal is not “as many tests as possible”; it is confidence in the critical paths.

If the suite feels slow or fragile, it usually means I am testing the wrong thing. In practice, I focus on the interfaces that operators actually rely on.

## What to test (a simple rule)

- **Unit tests**: small, fast checks for logic and pure functions.
- **Integration tests**: components + data + user flows (most value per test).
- **E2E tests**: a few happy-path checks across the whole app.
- **Protocol tests**: make sure command payloads and telemetry shapes stay stable.

## High-signal testing habits

- Prefer user-facing assertions (what the user sees/does).
- Use accessible queries first (roles, labels).
- Mock at the boundary (network) instead of mocking implementation details.
- Add one or two contract tests when a backend or device message format is shared by multiple clients.

## Things to avoid

- ~~Testing internal component state~~ when behavior is what matters.
- ~~Sprinkling test IDs everywhere~~ as a first choice.
- Overusing snapshots that fail for harmless UI changes.

## Wrap-up

Start small: one integration test for the most important flow, then add unit tests for tricky logic. Over time, the tests become living documentation for how the UI and hardware-facing pieces are supposed to behave.
