# REST API Design Principles I Use in Robotics and UAV Projects

When I build software around drones, robots, or internal tooling, the API has to stay boring in the best possible way: predictable, consistent, and easy to reason about. The more moving parts a system has, the more important it becomes to make the interface obvious.

This is the checklist I keep in mind when I am wiring together control software, telemetry, or device-facing tools.

## Core principles

- Use clear, consistent **resource names** (think nouns).
- Keep behavior aligned with **HTTP semantics** (read vs write).
- Return **consistent response shapes** so clients don’t guess.
- Prefer **sane defaults** with optional query parameters for filtering/sorting.
- Make error messages useful enough to debug field issues quickly.
- Keep device commands idempotent when the hardware can receive retries.

## A simple checklist

1. Pick stable resource paths (plural nouns are a common convention).
2. Use a small set of status codes consistently.
3. Document pagination and what “next/previous” means.
4. Be explicit about authentication and authorization requirements.
5. Add brief examples in docs for the “happy path” and common errors.
6. Separate telemetry endpoints from command endpoints so they do not drift into one another.
7. Version the contract before the hardware or client side forces a breaking change.

## Wrap-up

If you optimize for consistency first, your API will be easier to document, easier to test, and easier to keep stable when real hardware is involved.
