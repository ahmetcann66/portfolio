# TypeScript Patterns I Prefer in Control Software and Dashboards

TypeScript shines when it helps me model reality instead of fighting me all day. When I am building control software or a technical dashboard, a few small defaults make the codebase feel calmer, safer, and easier to refactor.

## Practical rules of thumb

- Turn on strictness and fix the sharp edges early.
- Prefer readable types over clever types.
- Use unions for “one of these”, interfaces for “shape of this”.
- Avoid `any` as a shortcut; it becomes ~~future debt~~ fast.
- Keep protocol payload types close to the code that serializes them.
- Model hardware states explicitly instead of using loose booleans.

## One pattern worth memorizing

```ts
type Result<T> =
  | { ok: true; value: T }
  | { ok: false; error: string };

export function parseNumber(input: string): Result<number> {
  const n = Number(input);
  return Number.isFinite(n) ? { ok: true, value: n } : { ok: false, error: "Not a number" };
}
```

That same shape works well for command acknowledgements, sensor parsing, and UI actions that can fail in more than one way.

## Wrap-up

The best TypeScript code reads like good documentation: clear names, predictable shapes, and errors that point me to the fix before a bug reaches hardware.
