# Git Workflow Habits for Hardware-Heavy Projects

Git is especially useful when a codebase touches firmware, control software, and operator tools. In those projects, I care less about fancy branching models and more about keeping history readable and reversible.

This is the workflow I prefer when a change can affect a real device.

## A simple team workflow

- Branch from `main` for every change.
- Keep branches small and short-lived.
- Open a pull request early and ask for review.
- Merge back to `main` once tests pass.
- Keep hardware-related changes isolated from UI and tooling changes when possible.
- Put calibration values and protocol changes in their own commits.

## Commit message tips

- Start with a clear verb (“add”, “fix”, “remove”, “refactor”).
- Keep the first line short and specific.
- Prefer multiple small commits over one giant “WIP”.
- Mention the subsystem when it matters: `control`, `vision`, `telemetry`, or `ui`.

## PR checklist

1. Explain what changed and why.
2. Link the issue or describe the user impact.
3. Add tests (or explain why not).
4. Keep the diff small enough to review quickly.
5. Call out any hardware assumptions, baud rates, pin mappings, or protocol versions.

## Wrap-up

The goal isn’t “perfect Git” — it’s fewer surprises when the software meets hardware. Start simple, write down the rules, and improve them as the system grows.
